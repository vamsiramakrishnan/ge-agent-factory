The core idea

How a coding agent becomes an agent-building system

A general coding agent produces free-form, non-deterministic output. Four control layers make its output deterministic and gated: harness, templating, pipeline checks, and critique.

The harness, templating, pipeline checks, and critique constrain the coding agent so its output is deterministic, schema-conformant, and gated — the conditions required to generate 363 agents reproducibly.

The raw material — how a spec is written

The spec is authored as code, inside the deck

A domain expert writes the use case as typed TypeScript objects on a presentation slide. A sync step extracts them into one machine catalog of 363 specs. The deck is the source of truth.

Load-bearing signal: a slide that defines no explicit generationSpec is synced with behaviorContract: null — a deliberate flag that the use case still needs enrichment before the factory will generate an ADK agent.

Phase 1 — manufacture each agent (×363)

The 14-station assembly line

Six work cells over one conveyor. Each station self-enqueues the next, so the line is durable and resumable — a worker crash just redelivers the task.

Bounded by targetStage · per-stage retry classes make redelivery reconcile, never duplicate19 / 21
What every agent runs on

Google’s agent stack, end to end

No bespoke runtime. Generated agents resolve tools from Agent Registry by name under their own agent identity — no hardcoded URLs, ADC supplies the credential.

One generic MCP image serves 363 tenants, each linked by name: the agent code references mcpServers/<id>, the registry resolves it to the dept URL ?agent=<id>, and the service loads that agent’s manifest and stores.
