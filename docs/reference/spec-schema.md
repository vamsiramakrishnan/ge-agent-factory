# Use-case spec schema

The use-case spec is the contract the whole factory reads from. A built sample
catalog lives at
[`apps/factory/generated/use-cases.generated.json`](https://github.com/vamsiramakrishnan/ge-agent-factory)
(git-ignored, regenerated by `make catalog`); the shape is defined by the emitter
[`apps/factory/scripts/emit-baseline-specs.mjs`](https://github.com/vamsiramakrishnan/ge-agent-factory)
and typed in `apps/presentation/src/types/architecture.ts`.

A spec entry has three top-level parts:

```jsonc
{
  "generationSpec":   { /* data shape + validation */ },
  "behaviorContract": { /* agent constraints, tools, workflow, evals */ },
  "architecture":     { /* system connections + pipeline stages */ }
}
```

Examples below are drawn from the `account-reconciliation-agent` entry.

---

## `generationSpec`

Describes the data the agent works over and how it is packaged.

| Field | Type | Notes |
|---|---|---|
| `version` | `1` | literal |
| `rowPolicy` | object | `{ defaultRowsPerEntity, minimumRowsPerEntity, seed, rationale }` |
| `sourceSystems[]` | object[] | `{ id, name, owns[], protocol, localBacking[], toolNames[], evidence[] }` |
| `entities[]` | object[] | `{ name, sourceSystemId, datastore, rowCount, primaryKey, columns[] }` |
| `relationships[]` | object[] | `{ from, to, cardinality, orphanPolicy }` |
| `documents[]` | object[] | audit-citable docs (see below) |
| `apis[]` | object[] | state-changing endpoints the agent calls |
| `anomalies[]` | object[] | seeded gaps for the agent to discover |
| `datastorePackaging` | object | where tables/docs live (alloydb / bigquery / cloudStorage / apis) |
| `validation` | object | `{ smokePrompt, expectedAnswer[], assertions[] }` |

**`sourceSystems[]`**

```json
{
  "id": "sap_s_4hana_fi",
  "name": "SAP S/4HANA FI",
  "owns": ["gl_entries", "subledger_balances", "open_items"],
  "protocol": "RFC/BAPI",
  "localBacking": ["alloydb"],
  "toolNames": ["query_sap_s_4hana_fi_gl_entries"],
  "evidence": ["source_system_record", "generated_audit_trail"]
}
```

**`entities[].columns[]`** — typed columns, e.g. `{"name":"amount","type":"float","min":-50000,"max":50000,"decimals":2,"required":true}`. Column types include `seq`, `date`, `enum` (with `values`), `float`/`number`, `boolean`, `ref:TABLE.COL`, and faker types.

**`documents[]`** — `type` is one of `policy | report | sop | contract | knowledge_base | audit | memo | guide | runbook | playbook`; carries `requiredSections[]`, `linkedEntities[]`, `minimumWordCount`, `citationAnchors[]`.

**`apis[]`** — `{ id, sourceSystemId, method, path, description, requestSchema, responseSchema, idempotencyKey }`.

## `behaviorContract`

The behavior contract is what turns a generic data agent into a task-specific one.
It is optional in the type, but **required at build time** for the factory to
generate a real agent.

| Field | Type | Notes |
|---|---|---|
| `role` | string | the agent's self-identity |
| `primaryObjective` | string | single-sentence success criterion |
| `inScope[]` / `outOfScope[]` | string[] | supported workflows / explicit refusals |
| `toolIntents[]` | object[] | the tools the agent may call (see below) |
| `evidenceRequirements[]` | object[] | `{ claim, mustCite[], sourceSystemIds[] }` |
| `escalationRules[]` | object[] | `{ trigger, action, handoffTarget, rationale }` |
| `refusalRules[]` | string[] | hard guardrails (PII, compliance) |
| `goldenEvals[]` | object[] | end-to-end eval cases (see below) |
| `workflow` | object | multi-agent topology (optional; see below) |
| `answerableQueries[]` | object[] | the questions the agent can answer (optional; derived if absent) |

**`toolIntents[]`** — `kind` is one of `query | action | evidence_lookup | notification | calculation`:

```json
{
  "name": "action_sap_s_4hana_fi_close",
  "kind": "action",
  "sourceSystemId": "sap_s_4hana_fi",
  "description": "Execute the close step after evidence is gathered and gates pass.",
  "requiredInputs": ["target_id", "rationale"],
  "produces": ["action_id", "audit_record_id"],
  "evidenceEmitted": ["api_response", "generated_audit_trail"]
}
```

`escalationRules[].action` is one of `escalate_to_human | refuse | request_more_info | use_fallback_tool`. `evidenceEmitted` values are `sql_result | source_system_record | document_reference | generated_audit_trail | api_response`.

### `behaviorContract.workflow`

The multi-agent topology, derived from `architecture.pipeline` + `toolIntents` by
`buildWorkflowFromPipeline` in
[`apps/factory/scripts/factory/agent-workflow.mjs`](https://github.com/vamsiramakrishnan/ge-agent-factory).
It is **only emitted when the pipeline has ≥2 tool-bearing stages spanning ≥2
distinct tools** — otherwise the agent is single-agent and `workflow` is omitted.

```json
{
  "mode": "sequential",
  "steps": [
    {
      "id": "balance__document__pull",
      "label": "Balance & Document Pull",
      "description": "Pull GL balances and sub-ledger details; match against BlackLine templates.",
      "tools": ["query_sap_s_4hana_fi_gl_entries", "query_blackline_reconciliations"]
    }
  ]
}
```

- **`mode`**: `sequential` | `parallel`. Parallel only when
  `architecture.parallel === true` or every stage is `parallel: true`; otherwise
  `sequential`.
- **`steps[]`**: `{ id, label, description, tools[] }`. The factory maps
  `sequential` → ADK `SequentialAgent`, `parallel` → `ParallelAgent`, with one
  sub-agent per step scoped to that step's `tools`.

### `goldenEvals[]`

```json
{
  "id": "account-reconciliation-agent-end-to-end",
  "prompt": "Run the workflow for the current period; cite evidence and surface escalations.",
  "expectedToolCalls": ["query_sap_s_4hana_fi_gl_entries", "action_sap_s_4hana_fi_close"],
  "mustReferenceEntities": ["gl_entries", "reconciliations"],
  "mustCiteDocuments": ["account-reconciliation-agent-controls-playbook"],
  "expectedActionOutcome": "Close executed against SAP S/4HANA FI with an audit-trail entry.",
  "forbiddenBehaviors": ["do not invent KPI numbers", "do not skip evidence_lookup"]
}
```

### `answerableQueries` and eval `mechanisms`

These are the two **capability concepts OKF cares about**. They may be authored
explicitly on the contract; if absent, they are **derived** by
[`apps/factory/scripts/lib/okf-capabilities.mjs`](https://github.com/vamsiramakrishnan/ge-agent-factory)
so older specs gain queries/tests without re-authoring:

- **`answerableQueries[]`** — `{ id, request, tools[], evidence[], stage? }`: the
  requests an agent can serve, each with the tools that serve it and the workflow
  stage it runs in. Derivation: explicit `answerableQueries` → else one per
  workflow **step** → else one per `inScope` item served by the read-tool surface
  (`deriveAnswerableQueries`).
- **eval `mechanisms`** — per golden eval, the tool names that **must** fire.
  Derivation: explicit `eval.mechanisms` → else the eval's `expectedToolCalls`
  (`evalMechanisms` / `deriveTestMechanisms`). Each derived test record carries
  `{ id, prompt, mechanisms[], rubric?, mustReferenceEntities, mustCiteDocuments, validates? }`,
  where `validates` links the test back to the Query Capability it exercises.

## `architecture`

| Field | Type | Notes |
|---|---|---|
| `connections[]` | object[] | `{ system, description, direction, protocol, category }` |
| `pipeline[]` | object[] | `{ label, description }` — the source for `behaviorContract.workflow` |

`connections[].direction` is `read | write | bidirectional`;
`category` is `erp | analytics | ai | clm | market-data | collaboration`.

`pipeline[]` is consumed at authoring time: `buildWorkflowFromPipeline` matches
each stage's label/description against the tool intents to produce the multi-agent
`workflow`.

> **Note:** the `account-reconciliation-agent` sample lives in
> `apps/factory/generated/use-cases.generated.json` (under the app dir),
> not at the repo root.
